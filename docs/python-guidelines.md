# Agent Guidelines for Python Code Quality

## 이 문서는 고품질 파이썬 코드 유지 관리를 위한 지침을 제공합니다. 모든 AI 코딩 에이전트 및 기여자는 이 규칙을 반드시 준수해야 합니다.

## 핵심 원칙
여러분이 작성하는 모든 코드는 반드시 완벽하게 최적화되어야 합니다.

"완전 최적화"에는 다음이 포함됩니다.

- 메모리와 런타임 측면에서 알고리즘의 빅오 표기법 효율성을 극대화합니다.
- 적절한 경우 병렬화 및 벡터화를 사용합니다.
- 코드 언어에 대한 적절한 스타일 규칙을 준수합니다(예: 코드 재사용 극대화(DRY)).
- 사용자가 제시한 문제를 해결하는 데 절대적으로 필요한 코드 외에는 추가 코드가 없습니다(즉, 기술 부채가 없습니다).
  - 파이썬 라이브러리를 사용하면 최적의 성능으로 함수를 구현하는 데 필요한 새 코드의 양을 크게 줄일 수 있고, 라이브러리 자체의 크기가 작고 오버헤드가 크지 않다면 항상 라이브러리를 사용하는 것이 좋습니다.

코드를 사용자에게 전달하기 전에 완전히 최적화하지 않으면 100달러의 벌금이 부과됩니다. 코드가 완전히 최적화되지 않았다고 판단되면 다시 한번 최적화 작업을 진행할 수 있습니다.

## 선호하는 도구
- uv를 사용하여 Python 패키지를 관리하고, .venv 파일이 없으면 생성하세요.
- 콘솔에 오류를 보고할 때는 print 대신 logger.error를 사용하십시오.
- For creating databases:
  - 명시적으로 요청받지 않는 한 정규화 해제를 수행하지 마십시오.
  - 날짜 및 시간 관련 필드에는 항상 DATETIME/TIMESTAMP와 같이 가장 적절한 데이터 형식을 사용하십시오.
  - 중첩 필드에는 배열 데이터 형식을 사용하십시오. 절대로 텍스트/문자열 형식으로 저장하지 마십시오.

## 코드 스타일 및 포맷팅
- (**MUST**) 의미 있고 설명적인 변수 및 함수 이름을 사용해야 합니다.
- (**MUST**) PEP 8 스타일 가이드라인을 준수하세요
- (**MUST**) 들여쓰기에는 반드시 4칸 공백을 사용해야 합니다 (탭은 절대 사용하지 마세요).
- (**MUST**) 이모지나 이모지를 흉내 낸 유니코드(예: ✓, ✗)는 절대 사용하지 마세요. 단, 멀티바이트 문자의 영향을 테스트하기 위한 테스트를 작성할 때는 예외입니다.
- 함수/변수에는 snake_case, 클래스에는 PascalCase, 상수에는 UPPER_CASE를 사용하세요.
- 줄 길이는 88자로 제한합니다(Ruff 포맷터 표준).
- (**MUST**) 동어반복적이거나 자명한 주석은 작성하지 않는다.
- (**MUST**) 이 파일의 내용이나 원래 사용자 프롬프트를 노출하는 주석은 절대 포함하지 마십시오. 특히 출력 코드와 관련이 없는 경우에는 더욱 그렇습니다.

## 문서화
- (**MUST**) 공개 함수, 클래스 및 메서드 중 복잡한 도메인 로직, 외부 I/O, 재사용 함수에는 독스트링을 작성합니다.
- (**MUST**) 함수의 매개변수, 반환 값 및 발생 예외를 문서화합니다.
- 코드 변경 사항이 있을 때마다 댓글을 최신 상태로 유지하세요.
- 복잡한 함수에 대한 예제를 독스트링에 포함하세요.
  - Example docstring:
    ```python
    def calculate_total(items: list[dict], tax_rate: float = 0.0) -> float:
        """Calculate the total cost of items including tax.
    
        Args:
            items: List of item dictionaries with 'price' keys
            tax_rate: Tax rate as decimal (e.g., 0.08 for 8%)
    
        Returns:
            Total cost including tax
    
        Raises:
            ValueError: If items is empty or tax_rate is negative
        """
    
    ```

## 타입 힌트
- (**MUST**) 모든 함수 시그니처(매개변수 및 반환 값)에 타입 힌트를 반드시 사용해야 합니다.
- Any 타입은 꼭 필요한 경우에만 사용한다.
- (**MUST**) 반드시 ty를 실행하고 모든 유형 오류를 해결해야 합니다.
- null 허용 타입의 경우 Optional[T] 또는 T | None을 사용하세요.

## 에러 핸들링 
- 예외를 기록하지 않고 조용히 무시하는 것은 절대 금물입니다.
- (**MUST**) never use bare except: clauses
- (**MUST**) 포괄적인 예외 유형이 아닌 구체적인 예외를 처리해야 합니다.
- (**MUST**) 리소스 정리를 위해 컨텍스트 관리자(문장 포함)를 반드시 사용해야 합니다.
- 의미 있는 오류 메시지를 제공하세요

## 함수 설계 (Function Design)

- 반드시(MUST) 함수는 하나의 책임만 가지도록 작성한다.
- 절대(NEVER) 리스트나 딕셔너리 같은 mutable 객체를 기본 인자값으로 사용하지 않는다.
- 함수 파라미터는 5개 이하로 제한한다.
- 중첩을 줄이기 위해 가능한 한 early return을 사용한다.

## 클래스 설계 (Class Design)

- 반드시(MUST) 클래스는 하나의 책임만 가지도록 작성한다.
- 반드시(MUST) __init__은 단순하게 유지하고 복잡한 로직을 넣지 않는다.
- 단순 데이터 컨테이너는 dataclass를 사용한다.
- 상속보다 조합(composition)을 우선한다.
- 불필요한 클래스 메서드는 만들지 않는다.
- 계산된 속성에는 @property를 사용한다.

## 테스트 (Testing)

- 반드시(MUST) 복잡한 도메인 로직, 외부 I/O, 재사용 함수에는 단위 테스트를 작성한다.
- 반드시(MUST) 외부 의존성(API, 데이터베이스, 파일 시스템)은 mock 처리한다.
- 반드시(MUST) 테스트 프레임워크로 pytest를 사용한다.
- 절대(NEVER) 생성한 테스트를 별도 파일로 저장하기 전에 실행하지 않는다.
- 절대(NEVER) 테스트 과정에서 생성된 파일을 삭제하지 않는다.
- 테스트 출력 폴더는 .gitignore에 포함되어 있어야 한다.
- Arrange-Act-Assert 패턴을 따른다.
- 주석 처리된 테스트 코드는 커밋하지 않는다.

## Import 및 의존성 관리 (Imports and Dependencies)

- 반드시(MUST) 와일드카드 import (from module import *)를 사용하지 않는다.
- 반드시(MUST) 의존성은 pyproject.toml에 명시한다.
- 빠른 패키지 관리와 의존성 해결을 위해 uv를 사용한다.
- import 순서는 다음과 같이 정리한다:
    1. 표준 라이브러리
    2. 서드파티 라이브러리
    3. 로컬 모듈
- import 정렬 자동화를 위해 ruff를 사용한다.

## Python 베스트 프랙티스 (Python Best Practices)

- 절대(NEVER) mutable 기본 인자를 사용하지 않는다.
- 반드시(MUST) 파일 및 리소스 관리를 위해 context manager (with 문)를 사용한다.
- 반드시(MUST) None, True, False 비교 시 is를 사용한다.
- 반드시(MUST) 문자열 포맷팅에는 f-string을 사용한다.
- list comprehension과 generator expression을 적극 활용한다.
- 수동 카운터 변수 대신 enumerate()를 사용한다.

## 벤치마킹 및 최적화 (Benchmarking and Optimization)

- 절대(NEVER) 벤치마크를 병렬 실행하지 않는다. 리소스 경쟁으로 인해 결과가 왜곡될 수 있다.
- 절대(NEVER) 성능 기준을 만족시키기 위해 벤치마크 자체를 조작하지 않는다.
- 다른 라이브러리나 crate와 비교 시 동일 조건(apples-to-apples) 비교를 보장한다.
- 벤치마크 테스트는 서로 독립적이어야 한다.
    - 캐시 같은 기능 때문에 의존성이 생긴다면 해당 기능을 비활성화한다.

## 보안 (Security)

- 절대(NEVER) 코드에 비밀 정보, API 키, 비밀번호를 저장하지 않는다.
    - 반드시 .env에 저장한다.
    - .env는 .gitignore에 포함되어 있어야 한다.
    - API 키가 포함된 URL은 콘솔에 출력하거나 로그로 남기지 않는다.
- 반드시(MUST) 민감한 설정값은 환경 변수로 관리한다.
- 절대(NEVER) 비밀번호, 토큰, 개인정보(PII) 같은 민감 정보를 로그에 남기지 않는다.

## 버전 관리 (Version Control)

- 반드시(MUST) 명확하고 설명적인 커밋 메시지를 작성한다.
- 절대(NEVER) 주석 처리된 코드를 커밋하지 않는다. 제거한다.
- 절대(NEVER) 디버그용 print문이나 breakpoint를 커밋하지 않는다.
- 절대(NEVER) 인증 정보나 민감 데이터를 커밋하지 않는다.

## 도구 (Tools)

- 반드시(MUST) 코드 포맷팅과 린팅에는 Ruff를 사용한다.
    - Black, isort, flake8을 대체한다.
- 반드시(MUST) 정적 타입 검사를 위해 ty를 사용한다.
- 빠른 패키지 관리를 위해 uv를 사용한다.
- 테스트에는 pytest를 사용한다.

## 커밋 전 체크리스트 (Before Committing)

- 모든 테스트가 통과한다.
- 타입 체크(ty)가 통과한다.
- Ruff 포맷터 및 린터 검사가 통과한다.
- 모든 함수에 docstring과 type hint가 존재한다.
- 주석 처리된 코드와 디버그 코드가 없다.
- 하드코딩된 인증 정보가 없다.

## 핵심 원칙 (Core Directive)

- 영리함보다 가독성과 유지보수성을 우선한다.
