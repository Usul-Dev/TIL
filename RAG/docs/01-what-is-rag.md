# RAG가 무엇인가

> 백엔드 엔지니어 관점에서 "RAG가 결국 무슨 시스템인지"를 먼저 잡는다.
> 도구(벡터 DB·임베딩 라이브러리)는 뒤에서 필요해질 때 붙인다.

## 한 줄 정의

**RAG (Retrieval-Augmented Generation)** = 질문이 들어오면 **관련 문서를 검색해서**,
그 문서를 **프롬프트에 끼워 넣고**, LLM이 **그 문서를 근거로 답을 생성**하는 방식.

검색(Retrieval)으로 보강한(Augmented) 생성(Generation). 이름이 곧 구조다.

## 왜 필요한가

LLM은 학습이 끝난 시점까지의 지식만 가지고 있고, 우리 사내 문서·최신 데이터는 모른다.
모델을 다시 학습시키지 않고 **외부 지식으로 답하게** 만드는 가장 단순한 방법이 RAG다.

백엔드 비유로 보면 RAG는 특별한 게 아니라 **익숙한 요청 처리 핸들러**다.

```
[일반 API]   요청 → DB 조회 → 응답 조립 → 반환
[RAG]        질문 → 문서 검색 → 프롬프트 조립 → LLM 응답 반환
```

지식이 **코드(모델) 밖**에 있고, 요청 때마다 읽어온다.
그래서 문서만 갈아 끼우면 지식이 갱신된다 — 재배포도, 재학습도 없다.

## 두 갈래의 흐름

RAG를 돌리려면 데이터를 다루는 흐름이 두 개로 나뉜다.

### 1. 인덱싱 (미리, 오프라인) — 백엔드의 ETL/마이그레이션에 해당

```
문서 로드 → 청킹(자르기) → 임베딩(벡터화) → 저장
```

- **청킹**: 문서가 너무 길면 프롬프트에 다 못 넣고 검색 정확도도 떨어진다. 그래서 문단 단위로 쪼갠다.
- **임베딩**: 텍스트를 숫자 벡터로 바꾼다. "의미가 비슷하면 벡터도 가깝다"는 성질 덕에 의미 기반 검색이 된다 (키워드 일치가 아니라 의미 일치).

### 2. 질의 (실시간, 요청 시)

```
질문 임베딩 → 벡터 검색(top-k) → 프롬프트 주입 → LLM 응답
```

질문도 같은 방식으로 벡터로 바꾼 뒤, 가장 가까운 문서 조각 k개를 찾아 프롬프트에 넣는다.

## "꼭 이 전부가 필요한가?" — 아니다

RAG의 **본질은 "검색 → 주입 → 생성"** 이 한 줄이다.
청킹·임베딩·벡터 검색은 **문서가 많아서 프롬프트에 다 못 넣을 때** 비로소 필요한 도구다.

- 문서가 몇 개 안 되고 짧다 → 임베딩·벡터 DB 없이 **전부 프롬프트에 넣어도** 된다. 더 단순하고 정확할 때도 많다.
- 리랭킹·하이브리드 검색 → 기본 검색 품질이 부족할 때 **나중에** 얹는 선택지.

> 그래서 이 디렉터리는 벡터 DB 없이 시작해서, "프롬프트에 다 안 들어간다"는 불편을
> 직접 겪은 뒤에 검색을 도입한다. 도구의 필요성을 몸으로 먼저 안다.
